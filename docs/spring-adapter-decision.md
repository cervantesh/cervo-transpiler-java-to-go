# Spring Adapter Decision

Date: 2026-06-16

Status: exploratory, not part of the core transpiler yet.

## Decision

Do not implement Spring lowering in the deterministic Java-to-Go core until pure Java library migration is more reliable.

Spring support should start as an adapter that reads project metadata, annotations, deterministic reports, and generated Go snippets. It should produce inventory, risk classification, and optional scaffolding. It should not silently translate framework behavior into Go business logic.

## Why

Spring code is not only Java syntax. It depends on runtime behavior:

- annotation scanning
- dependency injection and bean lifecycles
- proxy behavior
- HTTP routing
- validation
- configuration binding
- exception mapping
- transactions
- repository implementations generated by Spring Data

Translating that directly before the Java semantic engine is stronger would produce misleading Go.

## Evaluation Matrix

| Spring area | Initial handling | Rationale |
| --- | --- | --- |
| Annotations | Inventory and report | Annotation presence can be detected, but semantics depend on framework context. |
| Controllers | Generate optional HTTP route scaffolding | `@RestController`, `@GetMapping`, `@PostMapping`, and similar annotations can map to route skeletons after manual review. |
| Services | Migrate pure Java method bodies when supported; wire manually | `@Service` mostly marks lifecycle/DI, while method bodies may be ordinary Java. |
| Repositories | Advisory/manual by default | Spring Data generates implementations from interfaces and method names; direct Go translation is not equivalent. |
| Dependency injection | Generate constructor wiring notes or scaffolding | Go can use explicit constructors, but bean selection, scopes, proxies, and qualifiers need review. |
| Config properties | Generate config struct scaffolding | `@ConfigurationProperties` can become typed config structs if binding rules are explicit. |
| Exception handlers | Advisory/manual first | Java exceptions must map to explicit Go error returns and HTTP response handling. |
| Transactions | Advisory/manual | Spring transactions depend on proxies and runtime boundaries. |
| Validation | Advisory/scaffolding | Bean validation annotations can become validation notes or generated validators later. |

## Automation Policy

### Deterministic Automation

Allowed only when behavior is local and syntax/semantics are already supported:

- pure Java service methods with supported language features
- simple DTO-like classes after object model lowering is stronger
- config property structs when fields are plain supported types
- route inventory from controller annotations

### Scaffolding

Allowed when the adapter can generate a useful Go shell without claiming behavioral equivalence:

- HTTP route skeletons for controller methods
- constructor wiring for service dependencies
- config structs for `@ConfigurationProperties`
- handler placeholders for exception mappings

Generated scaffolding must be labeled as scaffolding and must point back to source files and annotations.

### Advisory/Manual

Required for framework behavior that is implicit, runtime-driven, or cross-cutting:

- Spring Data repositories and query derivation
- transactions
- security filters
- bean scopes and lifecycle hooks
- AOP/proxies
- exception-to-response semantics
- validation beyond simple field constraints

## Proposed Adapter Outputs

Future command shape:

```powershell
j2go spring scan ./spring-app --out build/spring-inventory.json
j2go spring plan ./spring-app --out build/spring-plan.md
```

Expected outputs:

- annotation inventory
- controller route list
- dependency graph hints
- repository/manual-risk list
- config property candidates
- exception handler inventory
- scaffolding recommendations

## Fixtures

Exploratory fixtures live in `explorations/spring/fixtures`. They are not part of the deterministic transpiler golden suite yet. Their purpose is to document Spring patterns that a future adapter must classify before any lowering is attempted.

## Exit Criteria Before Implementation

Before building a real Spring adapter:

- Phase 6 semantic model should resolve multi-file Java symbols more completely.
- Phase 8 migration should handle package imports and name collisions more deeply.
- Reports should include annotation inventory without requiring the core transpiler to support annotations as code generation input.
- The adapter must have separate tests proving it does not run during default `transpile` or `migrate`.
